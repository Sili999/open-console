RetroConsole OS

Modulare, futuristische Gaming-Konsole mit Fingerabdruck-Login auf Raspberry Pi 5
Raspberry Pi 5 · 4 GB RAM RetroPie OS · SD-Karte 5 Personen Team Grow R503 · WS2812B · Waveshare 5" Lizenz: MIT Open Source
1

Projektziel & Vision

Wir bauen eine modulare Open-Source Gaming-Konsole auf Basis des Raspberry Pi 5. Das Gerät vereint ein futuristisches Design mit einem klassischen Retro-Gaming-Erlebnis. Herzstück ist ein Fingerabdruck-Login-System: Jeder Nutzer legt den Finger auf den Sensor, die Konsole erkennt ihn, lädt sein persönliches Profil — inklusive Spielständen, Einstellungen und individueller RGB-Farbe — und startet EmulationStation direkt in seiner Umgebung. Der gesamte Quellcode wird auf GitHub veröffentlicht, sodass andere die Konsole nachbauen können.

System-Architektur

SchichtKomponentenTechnologie / Details
Application Auth-Modul · Retro-Emulatoren · System-Shell Fingerprint-Login · EmulationStation · SNES/GBA/PS1
OS / System Retro OS · User Manager · Driver Layer RetroPie (Pi 5) · ES --home pro User · Kernel-Module
Interface GPIO/SPI · USB/CSI · PWM/I2S Buttons & FP via GPIO · Webcam via USB · RGB via SPI
Hardware Raspberry Pi 5 · Input/Sensoren · Display & Power R503 · WS2812B · Waveshare 5" DSI · Powerbank

⚠ Kritische technische Abhängigkeiten
  • Pi 5 Config-Pfad: /boot/firmware/config.txt — nicht /boot/config.txt wie bei älteren Pi-Modellen
  • UART für R503: Bluetooth blockiert standardmäßig ttyAMA0 → dtoverlay=disable-bt muss gesetzt werden
  • RGB + Audio-Konflikt: WS2812B im PWM-Modus kollidiert mit Audio-DMA → zwingend SPI-Modus über GPIO 10 verwenden
  • R503 Spannung: Nur 3,3V — niemals direkt an 5V anschließen
  • EmulationStation --home: Jeder User bekommt sein eigenes Verzeichnis unter /home/pi/users/<id>/

Aufgabenstellung nach erfolgreicher Hardware-Verkabelung

Die folgenden Phasen beginnen, sobald alle Komponenten verdrahtet und am Pi angeschlossen sind. Jede Phase hat klare Verantwortlichkeiten und Abnahmekriterien. Phasen A–C laufen parallel — erst Phase D und später benötigen Ergebnisse aus mehreren Phasen gleichzeitig.

A

System-Grundkonfiguration

Voraussetzung für alle weiteren Phasen · zuerst abschließen
P2 Hardware P3 OS
RetroPie Pi 5 Image auf SD-Karte flashen
Offizielles RetroPie-Image für Pi 5 herunterladen und mit Raspberry Pi Imager auf SD-Karte schreiben. SSH vorab aktivieren (leere Datei ssh im Boot-Verzeichnis).
Kritisch
Waveshare 5" DSI-Display konfigurieren
In /boot/firmware/config.txt den Waveshare-spezifischen Overlay eintragen. Korrekte Zeile: dtoverlay=vc4-kms-v3d sowie das Waveshare-Overlay laut Hersteller-Dokumentation. Danach reboot und Display-Ausgabe verifizieren.
Kritisch
UART für Fingerabdrucksensor freischalten
In /boot/firmware/config.txt folgende Zeilen eintragen: enable_uart=1 und dtoverlay=disable-bt. Ohne disable-bt blockiert Bluetooth den ttyAMA0-Port. Danach: sudo systemctl disable hciuart und reboot.
Kritisch
SPI für RGB-LEDs aktivieren
Via sudo raspi-config → Interface Options → SPI → Enable. Verifizieren: ls /dev/spidev* muss /dev/spidev0.0 ausgeben. Nur SPI-Modus vermeidet den Audio/PWM-DMA-Konflikt auf dem Pi 5.
Hoch
SSH, WiFi und System-Update einrichten
WiFi-Credentials in /boot/firmware/wpa_supplicant.conf hinterlegen. Dann: sudo apt update && sudo apt upgrade -y. IP-Adresse notieren, damit alle Team-Mitglieder per SSH zugreifen können.
Hoch
Python-Libraries installieren
sudo pip install pyfingerprint rpi_ws281x --break-system-packages. Hinweis: rpi_ws281x muss mit sudo ausgeführt werden (SPI-Zugriff erfordert Root). RPi.GPIO ist bei RetroPie meist vorinstalliert.
Hoch
Abnahmekriterien Phase A
  • Display zeigt RetroPie-Bootscreen korrekt auf dem Waveshare 5" DSI
  • ls /dev/ttyAMA0 zeigt den UART-Port ohne Fehler
  • ls /dev/spidev0.0 zeigt den SPI-Port
  • SSH-Login von einem anderen Gerät im gleichen Netz funktioniert
B

Hardware-Komponenten testen

Parallel zu Phase A · alle Verbindungen verifizieren
P2 Hardware
Buttons via GPIO testen
Kurzes Python-Testskript schreiben: jeden Button-GPIO als Input mit Pull-Up konfigurieren (GPIO.setup(pin, GPIO.IN, pull_up_down=GPIO.PUD_UP)) und Tastendruck im Terminal ausgeben. Alle Buttons einzeln testen und GPIO-Nummern dokumentieren.
Hoch
WS2812B RGB-Streifen testen
Nach SPI-Aktivierung: sudo python3 set_rgb.py --mode flash --color red ausführen. LED-Streifen muss dreimal rot aufblinken. Falls keine Reaktion: SPI-Pin GPIO 10 und GND-Verbindung prüfen. Hinweis: set_rgb.py immer mit sudo ausführen.
Hoch
Fingerabdrucksensor R503 verbindungstest
Nach UART-Aktivierung: Sensor an TX/RX/VCC/GND anschließen. Test mit: python3 -c "from pyfingerprint.pyfingerprint import PyFingerprint; f=PyFingerprint('/dev/ttyAMA0',57600,0xFFFFFFFF,0x00000000); print(f.verifyPassword())". Ausgabe muss True sein.
Kritisch
Audio-Output testen
aplay /usr/share/sounds/alsa/Front_Center.wav. Wenn kein Ton: sudo raspi-config → System Options → Audio auf den richtigen Output setzen. Bei Audio+RGB-Konflikt: prüfen ob /dev/spidev0.0 statt PWM verwendet wird.
Mittel
Webcam-Erkennung prüfen
lsusb muss die Webcam auflisten. Für einen Schnelltest: sudo apt install fswebcam && fswebcam test.jpg. Bild muss in test.jpg gespeichert werden.
Optional
Komponentenmaße für Case-Person dokumentieren
Exakte Position und Maße aller verbauten Komponenten (Display, Buttons, FP-Sensor, Webcam, USB-Ports) aufmessen und an Person 1 weitergeben. Besonders: Abstand der DSI-Buchse zum Pi-Board und Powerbank-Anschlussposition.
Hoch
Abnahmekriterien Phase B
  • Alle Buttons lösen messbar GPIO-Events aus
  • RGB-Streifen blinkt auf Kommando in der richtigen Farbe
  • R503 antwortet mit True auf verifyPassword() — das ist die Mindestvoraussetzung für Phase C
  • Maßliste an Person 1 übergeben
C

Fingerabdruck-Auth-System implementieren

Startet sobald R503 via UART antwortet (Phase B)
P4 Auth
enroll.py ausführen — alle Team-Finger registrieren
sudo python3 enroll.py für jede Person ausführen. Das Skript gibt die Slot-ID aus (0, 1, 2, …). Slot-IDs notieren — sie werden in user_map.json eingetragen. Jede Person sollte den Zeigefinger des dominanten Handgelenks verwenden.
Kritisch
user_map.json befüllen
Für jede Person: Slot-ID → Name + RGB-Farbe (als [r,g,b]-Array) + Home-Pfad eintragen. Die Farbe wird später für die RGB-LEDs verwendet. Beispiel: "0": {"name": "Alex", "color": [0, 120, 255], "home": "/home/pi/users/0"}
Kritisch
User-Verzeichnisse anlegen
Für jede Slot-ID: mkdir -p /home/pi/users/0/.emulationstation. Das .emulationstation-Unterverzeichnis muss existieren, bevor ES gestartet wird. Hier werden Spielstände, Themes und Einstellungen pro User gespeichert.
Hoch
scan_finger.py testen — Scan und Matching prüfen
sudo python3 scan_finger.py ausführen, Finger auflegen. Ausgabe muss die korrekte Slot-ID (0, 1, …) sein. Unbekannten Finger testen → Exit-Code 3 erwartet. Timeout testen: 5 Sekunden warten → Exit-Code 2.
Hoch
login.sh implementieren und manuell testen
Skript ausführen: bash login.sh. Erwarteter Ablauf: RGB pulsiert weiß → Finger auflegen → RGB wechselt zur User-Farbe → EmulationStation startet mit --home /home/pi/users/<id>. ES beenden → Loop startet neu.
Kritisch
Autostart via systemd einrichten
Service-File nach /etc/systemd/system/retro-login.service erstellen. Wichtige Felder: Type=simple, ExecStart=/bin/bash /home/pi/login.sh, Restart=always, User=pi. Dann: sudo systemctl enable retro-login && sudo systemctl start retro-login.
Hoch
Fallback-Login implementieren (Button-Kombo)
In login.sh: Falls nach 5 Sekunden kein Finger erkannt, eine Button-Kombination abfragen (z.B. zwei Buttons gleichzeitig). Ermöglicht Login ohne Sensor bei Hardware-Problemen. GPIO-Nummern von Person 2 übernehmen.
Mittel
Abnahmekriterien Phase C
  • Alle 5 Team-Finger werden korrekt erkannt und ihrer Slot-ID zugeordnet
  • Unbekannte Finger → RGB rot, kein Login
  • Nach Login startet ES mit dem korrekten User-Home-Verzeichnis
  • Nach ES-Exit startet der Login-Loop automatisch neu
  • Systemd-Service überlebt einen Reboot (sudo reboot → Login-Screen erscheint automatisch)
D

RetroPie & EmulationStation konfigurieren

Parallel zu Phase C · benötigt fertiges OS aus Phase A
P3 OS P5 UI
ROMs in korrektes Verzeichnis kopieren
ROMs per SSH oder USB-Stick nach ~/RetroPie/roms/<system>/ kopieren. Unterstützte Systeme: snes, gba, psx, nes, megadrive. Mindestens 2–3 Spiele pro System für die Demo vorbereiten.
Hoch
EmulationStation manuell mit --home testen
emulationstation --home /home/pi/users/0 direkt im Terminal starten. ES muss starten und die ROMs aus dem globalen ROM-Verzeichnis anzeigen. Spielstände werden in /home/pi/users/0/.emulationstation/ gespeichert. Kritisch für das Profile-System.
Kritisch
Futuristisches ES-Theme installieren
Empfehlung: Alekfull NX oder Carbon Flat. Installation via RetroPie-Setup-Skript: sudo ~/RetroPie-Setup/retropie_setup.sh → Manage Packages → Themes. Theme pro User-Profil in der ES-Konfiguration setzen — so hat jeder User sein eigenes Theme.
Hoch
Controller-Mapping konfigurieren
Buttons-GPIO-Mapping in RetroArch / EmulationStation eintragen. Alternativ: mk_arcade_joystick_rpi-Kernel-Modul für GPIO-Buttons verwenden. GPIO-Nummern von Person 2 übernehmen. Jede Button-Funktion (A/B/Start/Select) testen.
Hoch
Boot-Zeit optimieren
Nicht benötigte Services deaktivieren: sudo systemctl disable bluetooth triggerhappy. Splash-Screen aktivieren: in /boot/firmware/cmdline.txt quiet splash hinzufügen. Ziel: unter 30 Sekunden vom Einschalten bis zum Login-Screen.
Mittel
Abnahmekriterien Phase D
  • ES startet mit --home und zeigt ROMs korrekt an
  • Mindestens 1 Spiel pro System lässt sich starten und spielen
  • Controller-Buttons funktionieren im Spiel
  • Futuristisches Theme ist aktiv und auf dem DSI-Display gut lesbar
E

RGB-Effekte & futuristische UI

Parallel zu C & D · integriert mit Phase C am Ende
P5 UI
set_rgb.py fertigstellen — alle Modi implementieren
Modi: off, solid --user <id>, flash --color red, pulse. Farben aus user_map.json einlesen statt hart kodieren: json.load(open("user_map.json"))[user_id]["color"]. So bleibt das System erweiterbar.
Hoch
Login-RGB-Effekt finalisieren
Sequenz: Weiß pulsieren (warten) → Finger erkannt → kurzer weißer Blitz → sanfter Übergang zur User-Farbe → User-Farbe pulsiert einmal → bleibt in der User-Farbe stabil während ES läuft → LEDs aus nach ES-Exit. Übergangs-Timing: ca. 0,8 Sekunden.
Kritisch
Futuristischen Splash-Screen erstellen
Splash-Image als PNG (Auflösung des Waveshare 5": 800×480 oder 1280×800 prüfen). Design: dunkler Hintergrund, Konsolen-Logo, ggf. Scanlines-Effekt. Splash aktivieren via sudo raspi-config → Display → Splash screen oder Plymouth-Splash.
Mittel
UI-Sounds einbinden
Kurze Sound-Effekte (.wav, unter 1 Sekunde) für: Login-Erfolg, Login-Fehler, ES-Start. Abspielen via aplay sound.wav & parallel zum RGB-Effekt in login.sh. Sounds im futuristischen Stil (Sci-Fi-Beeps, keine Retro-Chiptunes für den Login-Screen).
Mittel
Demo-Flow und GitHub README vorbereiten
README.md mit: Projektbeschreibung, Komponenten-Liste, Verdrahtungsdiagramm, Setup-Anleitung (Phase A–C als Schritte), Screenshot des laufenden Systems. Lizenz: MIT. Repo-Struktur: /scripts, /config, /docs, /media.
Hoch
Abnahmekriterien Phase E
  • Jeder User hat eine erkennbar andere RGB-Farbe beim Login
  • Unbekannter Finger → LEDs blinken rot
  • Login-Sound ist zu hören und zeitlich mit RGB synchronisiert
  • Splash-Screen erscheint beim Boot statt Kernel-Textausgabe
F

Integration & End-to-End-Test

Alle Phasen A–E müssen abgeschlossen sein
P2 P3 P4 P5
Vollständigen Login-Loop für alle 5 User testen
Jeden der 5 Finger nacheinander testen. Erwartetes Verhalten: RGB weiß → Scan → RGB User-Farbe + Sound → ES mit korrektem Profil → Spiel starten → ES beenden → RGB aus → RGB weiß → nächster User. Jeden Schritt abhaken.
Kritisch
Spielstand-Isolation prüfen
User 0 startet ein Spiel, speichert, beendet ES. User 1 loggt sich ein → darf den Spielstand von User 0 nicht sehen. Spielstände liegen in /home/pi/users/<id>/.emulationstation/, nicht im globalen Pfad. Isolation durch --home Flag sicherstellen.
Hoch
Reboot-Test: System ohne Maus/Tastatur bedienbar
Pi kalt starten (Powerbank ein). Ohne externe Geräte muss die Konsole automatisch in den Login-Screen starten. Alle Funktionen nur mit den eingebauten Buttons und dem Fingerabdrucksensor erreichbar sein.
Kritisch
Stresstest: 10x Login/Logout hintereinander
Den Login-Logout-Zyklus 10 Mal wiederholen. Login-Loop darf nicht abstürzen oder hängen. Systemd-Service muss nach jedem ES-Exit zuverlässig neu starten. Falls Probleme: journalctl -u retro-login -f für Live-Logs.
Hoch
Abnahmekriterien Phase F — Demo-Ready
  • Alle 5 User können sich einloggen und ein Spiel starten
  • Spielstände sind pro User isoliert
  • System startet nach Reboot selbstständig ohne externe Eingabegeräte
  • 10x Login/Logout ohne Absturz — das ist das Mindest-Kriterium für die Demo
G

Case-Montage & Demo-Vorbereitung

Person 1 durchgehend · finale Zusammenführung
P1 Case P5 UI
Alle Komponenten ins Case einbauen
Pi, Display, Powerbank, FP-Sensor, Buttons, RGB-Streifen und Webcam fest im Gehäuse montieren. Besonders: Kabel so verlegen dass keine Verbindungen unter Druck geraten. FP-Sensor muss von außen zugänglich und gut erreichbar sein (Demo!). RGB-Streifen entlang des Case-Rahmens.
Hoch
Abschluss-Test im zusammengebauten Zustand
Nach der Montage alle Phase-F-Tests wiederholen. Besonders: Hitze-Check nach 20 Minuten Betrieb (Pi 5 wird warm), Kabel-Zugtest, Button-Erreichbarkeit. Falls der Pi wärmer als 70°C wird: Lüftungsschlitze im Case prüfen oder kleinen Lüfter ergänzen.
Kritisch
Demo-Script für die Präsentation festlegen
Ablauf festlegen: (1) Einschalten + Splash zeigen, (2) Person 1 loggt sich ein → RGB Farbe 1, (3) Spiel kurz spielen, (4) Logout, (5) Person 2 loggt sich ein → andere RGB Farbe, (6) GitHub-Repo auf Laptop zeigen. Max. 3 Minuten. Alle 5 Personen kurz durch die Demo führen.
Hoch
GitHub-Repository veröffentlichen
Repo public schalten. Alle Skripte (login.sh, scan_finger.py, set_rgb.py, enroll.py, user_map.json als Template) committen. MIT-Lizenz-Datei hinzufügen. README mit Foto der fertigen Konsole ergänzen.
Mittel
Abnahmekriterien Phase G — Finales Produkt
  • Konsole ist vollständig verbaut, keine losen Kabel sichtbar
  • FP-Sensor von außen gut erreichbar
  • System läuft 20+ Minuten ohne Überhitzung oder Absturz
  • Demo-Ablauf sitzt — alle 5 Personen haben ihn einmal durchgespielt
  • GitHub-Repo ist öffentlich und enthält alle Skripte + README
